ovim — Agent Edit UX Exploration

Five approaches to giving users visibility and control when an AI agent modifies their code through the chat panel.

Approach A
Edit Summary in Chat
After the agent completes a turn that involved edits, inject a collapsible summary bubble into the conversation showing which files were modified, how many lines changed, and offering quick actions. Lightest-touch approach — no new UI surfaces, just richer chat content.
Zero new UI — lives in existing chat flow
Natural reading order: explanation → changes
Collapsible — doesn't clutter long conversations
Summary appears after the fact
No inline visibility in the buffer itself

Approach B
Gutter Markers
Show colored markers in the line number gutter indicating which lines the agent added, modified, or deleted — like git diff signs but specifically for agent edits. Clears when the user saves or accepts.
Immediate spatial awareness of what changed
Familiar pattern (git gutter signs)
Works for any buffer, even background ones
Needs signcolumn space (1 char)
Doesn't show what the old content was
Only visible when viewing that buffer

Approach C
Modified Files Bar
A thin status strip between the buffer and chat panel showing which files the agent has modified. Each file is a pill/badge that can be selected to jump to that buffer. Shows +/- counts. Always visible during a chat session with edits.
Persistent visibility of all modified files
Quick navigation to any modified buffer
Compact — just 1 row
Takes 1 line of vertical space
Clutters the layout when no edits have been made

Approach D
Diff Review Mode
A dedicated vim-like diff view activated by a keybinding (e.g. [Enter] on the changes bubble, or :ai diff). Shows side-by-side or unified diff of agent changes. User can accept (:w) or reject (u) per file. Exits back to chat on close.
Full context: exactly what changed, before and after
Familiar to vim users (vimdiff-style)
Enables informed accept/reject decisions
Interrupts the chat flow — a mode switch
Complex to implement (diff rendering)
May be overkill for small changes

Approach E
Composable: Summary + Gutter + Bar
Combine the best parts: a summary bubble in chat (A), gutter markers in the buffer (B), and a thin modified-files strip (C). Each layer adds information at a different zoom level — conversation context, spatial awareness, and cross-file navigation. The diff mode (D) is available on demand but not the default path.
Information at every zoom level
Each piece is independently useful
Gutter + bar can be toggled off
Diff review available but not forced
Most implementation work
Risk of visual clutter if all layers active

Recommended Path

Start with A (summary bubble) — it's the highest-value, lowest-cost change. The agent already returns tool results; surfacing a "Changes" summary at the end of a turn that involved edits is mostly a rendering concern. It gives the user immediate awareness of what happened without any new modes or UI surfaces.

Then add B (gutter markers) — the spatial "what changed where" awareness is the single most useful thing when reviewing edits in the buffer. This is a well-understood pattern (git gutter) that vim users already expect. It also naturally handles background buffers: when the user switches to a file the agent modified, the markers are already there.

C (modified files bar) and D (diff review) are valuable but secondary. The files bar matters most for multi-file edits, which won't be the common case initially. Diff review is powerful but complex — and less urgent because undo already provides a safety net. Both can be added later without changing the A+B foundation.

Key implementation insight: all four approaches share one prerequisite — tracking which lines/ranges were modified by the agent per-buffer. Build that tracking layer first (a Vec<(usize, usize, EditKind)> per buffer for agent-touched ranges), and A, B, C, D all become rendering decisions on top of the same data.
